# 任务计划：UR5 真实低维状态与带时间戳的多模态数据记录

## 目标
实现一套更可靠的 UR5 示教数据记录流程：记录真实的机器人低维状态、实际发送给 UR5 的关节控制指令、各模态独立时间戳，以及足够的元数据，用于把较慢的图像流和较快的机器人/控制流对齐，服务后续模仿学习。

## 关键问题
1. `control` 是否继续表示发送给 `env.step()` 的目标关节位置，还是也要单独保存 raw GELLO action？
2. 训练主时间轴应该锚定在 `robot_state_time`、`action_sent_time`，还是 `frame_save_time`？
3. 图像是否每个控制帧都保存一份缓存副本，还是用 frame id / 时间戳引用以避免大量重复图像？
4. `ee_pos_quat` 是否全仓库统一为 `[x,y,z,qw,qx,qy,qz]`，还是新增字段名把四元数顺序写清楚？
5. 现有下游脚本是原地适配，还是为当前多模态数据集新增转换脚本？

## 已做决定
| 决定 | 理由 |
|------|------|
| 将当前 `control` 视为 UR5 的目标关节命令 | 当前控制循环在调用 `env.step(action)` 前保存裁剪后的 action，因此它位于真实命令路径上。 |
| 保持改动外科手术式、尽量小 | 当前需求是数据真实性和同步，不是重写机器人控制架构。 |
| 用显式元数据替代隐含假设 | 模仿学习需要单位、来源、时间语义，避免在错误标签上训练。 |
| 不为了等待 30 Hz 相机而阻塞控制循环 | D405 配置为 30 FPS，而控制循环目标是 100 Hz；阻塞相机会降低控制质量。 |

## 备注
- 当前可信低维状态主要是 `joint_positions` 和 `force`；当前 `joint_velocities`、`ee_pos_quat`、无夹爪 UR5 下的 `gripper_position` 语义错误或只是占位。
- `control` 单位是 rad，默认应保持为实际发送的目标关节角；如需调试可额外保存 raw GELLO action。
- 传感器同步应通过显式时间戳表达，而不是因为数据在同一个 `.pkl` 文件里就假设严格同步。
